## ⭐ CEO 최신 지시 (2026-05-25) — "부모 먼저, 자녀는 나중"

> **"교육 대상은 자녀 먼저 보다는 더 관심이 많은 부모(주로 엄마)를 대상으로 하는 교육에 더 중점을 둬. 자녀는 부모가 충분히 AX에 대해 이해가 된 후에 다음 단계로 하는게 더 바람직해."**
>
> — 대표(CEO), 2026-05-25

### 이것이 바꾸는 것

**기존 (암묵적):**
```
타겟 = 학부모
제품 = 자녀 AI 사용 관련 가이드 (부모는 구매자, 자녀가 교육 대상)
```

**변경 후 (CEO 확정):**
```
타겟 = 부모 본인 (특히 엄마)
제품 = [1단계] 부모 본인 AX(AI 전환) 이해 교육
       [2단계] 부모가 이해한 후 → 자녀 교육 설계 가이드

순서: 부모 먼저 → 자녀는 그 다음
```

### 왜 이것이 더 올바른가

1. **비행기 산소마스크 원칙:** 부모가 먼저 산소마스크를 써야 아이를 도울 수 있다. AI 교육도 마찬가지.

2. **부모가 AI를 모르면 기준을 줄 수 없다:** "얼마나 써야 괜찮은가"를 부모 본인이 체험 없이 자녀에게 기준을 줄 수 없다.

3. **리텐션 메커니즘 자연 해결:** 부모 AX 커리큘럼 4~8주 + 자녀 교육 설계 4~8주 = 총 16주 구독이 자연스럽게 발생.

4. **진짜 개인화 가능:** 부모 본인 직업·AI 사용 수준·자녀 학년의 조합이 모두 다름 = Segment of One의 자연스러운 근거.

5. **경쟁자 없는 포지션:** 모든 경쟁자가 "자녀를 위한 AI 교육"을 판다. Harness만이 "부모 본인 AX 이해"를 먼저 판다.

### 커리큘럼 구조 (개정)

```
[1단계: 부모 AX 이해 — 4주]
  1주차: AI가 실제로 뭘 잘하고 뭘 못 하는가 (현실 인식)
  2주차: 부모 본인 업무에 AI 직접 써보기 (체험)
  3주차: AI가 인간 인지에 미치는 영향 (cognitive offloading 이해)
  4주차: 우리 가정의 AI 사용 기준 초안 잡기

[2단계: 자녀 교육 설계 — 4주 (1단계 완료 후)]
  5주차: 내 아이의 AI 사용 패턴 관찰하기
  6주차: 학년·성향별 AI 사용 기준 확정하기
  7주차: 자녀와 AI에 대해 대화하는 법
  8주차: 자녀 AI 의존 신호 감지 및 대응법
```

## 0. 한 문장 요약

> 시장의 대부분은 "자녀에게 AI를 어떻게 쓰게 할지"를 팔지만, Harness는 **"부모 본인이 AI를 먼저 이해하고, 그 이해를 바탕으로 자녀를 가이드하는 법"**을 판다 — 부모의 직접 체험 없이 자녀 교육은 공허하다.

---

## 1. 문제 정의 (Problem)

대표님 발의 내용을 사업 언어로 정리:

1. **전 세대 AI FOMO** — 특히 어릴수록, 그리고 그 자녀를 키우는 부모일수록 "뒤처질까" 하는 불안이 크다.
2. **막연한 불안만 있고 실행 지침이 없다** — 가이드·컨설팅 수단이 빈약.
3. **부모가 먼저 알아야 하는데 못 한다** — 부모의 AI 활용은 "네이버/구글 검색" 수준에 머묾.
4. **현재 학생의 AI 활용은 의존을 키운다** — 초중고: 과제 정답 찾기 / 대학생: 클로바노트류 강의요약·퀴즈 생성. → 주도적 학습에서 멀어지고 뇌 사용 빈도↓ → **AI 종속**.
5. **세대별 맞춤 컨설팅·교육이 절실하나 공급이 없다.**

→ **핵심 통증(pain): "내 아이(또는 나)가 AI를 쓸수록 오히려 멍청해지는 것 아닌가"라는 불안.** 이것이 결제를 만드는 감정이다.

## 2. 시장 결론 (Market — 상세는 EDU_MARKET_RESEARCH.md)

- 글로벌 AI 교육 시장은 고성장(CAGR 31~43%, `vendor-estimate`)이나, 절대 규모 추정치는 출처별 1.6배 편차 → **방향성만 신뢰**.
- 한국: 생성형 AI 사용 저변 급확대(국민 44.5% 경험), "쓸 줄 안다 ≠ 잘 쓴다" 간극이 컨설팅 수요의 토대.
- 경쟁 지형:
  - 대형 플레이어(엘리스 395억, 코드잇 64억/분기, 패스트캠퍼스)는 **B2B 출강 + B2C 구독/부트캠프**로 수익. 모두 **"도구 사용법·직무 전환"에 집중**.
  - 부모 타겟 상품(MissyShop류)은 이미 존재 → **수요 검증됨**. 단, 패키지 콘텐츠 판매에 가깝고 **지속 자문·반의존 학습은 미충족**.
  - 무료 AI 리터러시 toolkit(OpenAI·Common Sense)이 풍부 → **단순 소개는 차별화 불가, 판단·전략·맞춤화에서만 유료 가치 발생**.
- **빈틈(white space): "반(反)의존 학습법 × 지속형 부모 자문 × 한국 학부모 맥락" 교집합은 비어 있다.**

```
  → +7일: "다음 Step 준비됐어요" + 전환 버튼
```

## 6. Pilot 설계 (시범 운영)

> 목표: **최소 비용으로 "부모가 돈을 내는가"를 검증.** 인프라가 아니라 결제 신호가 목표.

## 8. 서비스 채널 로드맵 — CEO 2026-05-25 확정

> CEO 제안: "앱과 웹사이트를 동시 운영하고 카카오톡은 알림 등 보조 수단으로 사용."
> **비서실장 평가:** 장기 방향은 맞으나 Phase 1 앱 선개발은 Product-over-Pipeline Rule 위반 위험. Phase별 단계 진입 기준을 정의한다.

### 8.1 채널 벤치마크 (사주·운세 서비스 유사 사례)

| 서비스 | Phase 1 채널 | 스케일업 채널 | 특이점 |
|---|---|---|---|
| 점신 | 앱+웹 병행 | 앱 중심 | 앱이 메인 리텐션 도구 |
| 카카오 채널 역술인 | 카카오 채널 단독 | 채널 확장 | 채널만으로 수십만 팔로워 달성 사례 |
| 인스타+카카오 운세 | 인스타 → DM → 카톡 | 웹 랜딩 추가 | 앱 없이도 초기 유료 전환 가능 |

→ **Phase 1에서 앱 없이도 유료 전환 검증 가능.** 앱은 리텐션이 검증된 뒤 투자.

### 8.2 Phase별 채널 전략

| Phase | 진입 기준 | 채널 구성 | 예상 비용 |
|---|---|---|---|
| **Phase 1** | 현재 (Pretotyping) | 노코드 랜딩(Framer/Carrd) + 카카오 채널 + 구글폼 | $0~20/월 |
| **Phase 2** | 유료 전환 ≥ 10명 | PWA(Next.js/Vercel) + 카카오 채널 자동화 | $10~30/월 |
| **Phase 3** | 유료 구독자 ≥ 50명 | iOS/Android 네이티브 앱 + 웹 병행 + 카카오 보조 | 개발비 별도 |

**Phase 1 — 노코드 랜딩 + 카카오 채널**

```
[랜딩 페이지] Framer 또는 Carrd (노코드)
  → 결제 버튼 클릭 → WTP 시그널 측정 (Pretotyping CTR ≥ 2%)
  → 구글폼 온보딩 (자녀 학년, AI 사용 현황, 부모 AX 수준)

[카카오 채널]
  → Step 완료 알림, 3일/7일 리마인더, CS 문의 채널
  → 초기에는 수동 발송도 허용 (자동화는 Phase 2)
```

> ⚠️ **Phase 1 앱 선개발 금지:** Product-over-Pipeline Rule. 앱 개발 공수 = 첫 유료 구독자 1명 전에 투입 불가. 첫 결제 신호 확인 후 Phase 2에서 PWA 우선.

**Phase 2 — PWA (앱 없이 앱 같은 UX)**

```
PWA(Progressive Web App): 설치 불필요, 모바일 홈화면 추가 가능
  → Step별 진행 트래킹, 개인화 대시보드
  → 카카오 채널 알림 자동화 (Step 완료 → 즉시 알림)
  → 결제 연동: 토스페이먼츠 또는 카카오페이
```

**Phase 3 — 네이티브 앱 + 웹 병행 (CEO 제안 실현)**

```
iOS + Android 앱: 푸시 알림, 오프라인 Step 저장, 리텐션 극대화
웹(PC/모바일): 결제 랜딩, 관리자 대시보드
카카오 채널: 알림/CS/리마인더 보조 (앱 미설치 유저 대응)
```

### 8.3 채널별 역할 정의

| 채널 | 주요 역할 | 도입 시점 |
|---|---|---|
| 노코드 랜딩 | Pretotyping, WTP 클릭 측정 | Phase 1 즉시 |
| 카카오 채널 | 알림, CS, 리마인더 | Phase 1 즉시 |
| 구글폼/노션 | 온보딩 수집, 피드백 | Phase 1 즉시 |
| PWA(Next.js) | Step 트래킹, 개인화 UX | Phase 2 진입 시 |
| 네이티브 앱 | 풀 UX, 푸시 알림, 리텐션 | Phase 3 진입 시 |
| 웹 PC | 결제 랜딩, 대시보드 | Phase 2~ |

## 9. VP 리뷰어 풀 확장 설계 — CEO 2026-05-25 지시

> CEO 지시: "향후 본격 서비스가 이루어진다면 전문 리뷰어군을 모집하여 온라인으로 의견을 다는 식으로 하면 될 거 같아. VP가 다 하기에는 어려워."

### 9.1 배경 및 현재 VP SLA 한계

현재 VP(부대표)는 Step 검수 10%를 담당. 구독자 50명 초과 시 일 50건 도달 → VP 단독 SLA 위협.

| 구독자 수 | 일 검수 예상 건수 | VP 단독 가능 여부 |
|---|---|---|
| 0~20명 | ≤ 20건 | ✅ 가능 (일 60분 이내) |
| 20~50명 | 20~50건 | ⚠️ 경계 |
| 50명+ | 50건+ | ❌ VP 단독 불가 → 리뷰어 풀 필요 |

### 9.2 Phase별 리뷰어 풀 운영 구조

| Phase | 운영 구조 | 리뷰어 수 |
|---|---|---|
| Phase 1 (0~50명) | VP 단독 | 1명 |
| Phase 2 (50~200명) | VP(Quality Lead) + 외부 리뷰어 | 3~5명 |
| Phase 3 (200명+) | VP 리드 + 리뷰어 풀 확장 | 10명+ |

### 9.3 리뷰어 자격 기준

**필수 조건:**
- 초중등 학부모 경험 보유 (현역 부모 우선)
- AI 도구 기본 사용 경험 (ChatGPT, 클로바노트 등)
- 비전문가 관점에서 콘텐츠 자연스러움 평가 가능

**비추천 프로파일:**
- 교육전문가·교사·학원 강사 → "전문가 처방" 인식 유발 → 법무 카피 리스크 증가
- AI 헤비유저 → 일반 부모 관점 이탈 가능성

### 9.4 리뷰 방식 (온라인)

```
[리뷰 체크리스트 — 건당 5분 목표]
① 자연스러운가? (어색한 표현, 불필요한 영어 용어)
② 이해 가능한가? (비전문가 부모가 읽어도 이해됨)
③ 실행 가능한가? (이번 주 실제로 할 수 있는 내용)
④ 불쾌하거나 부담스러운 표현이 있는가?
⑤ 공유 가능한가? ("지인에게 보낼 만하다" 여부)

[반려 시 필수: 구체적 피드백 메모 (1~2줄)]
```

**도구:** 노션 대시보드 또는 구글폼 (Phase 2 기준)

### 9.5 보상 체계

| Phase | 보상 형태 | 세부 내용 |
|---|---|---|
| Phase 2 초기 | Step 무료 제공 | 리뷰 10건 완료 → Step 1개 무료 |
| Phase 2 중기 | 소액 리뷰비 | 건당 ₩300~500 (capital_action_approve 후 확정) |
| Phase 3 | 월 고정 계약 | 20~30건/주 기준 ₩50,000~100,000/월 (검토 중) |

> ⚠️ **소액 리뷰비 도입 시:** `capital_action_approve` 필요 (실제 금전 지급).
> ⚠️ **외부 리뷰어 공개 모집 시:** `legal_review_approve` 필요 (프리랜서 vs 근로계약 분류).

### 9.6 온보딩 프로세스

```
1. 리뷰어 지원 (간단한 자기소개 폼)
2. 샘플 Step 2건 무료 리뷰 테스트
3. VP 검토 → 통과 시 정식 풀 등록
4. 리뷰어 가이드 제공 (체크리스트 + 금지 표현 목록)
5. 월 1회 품질 평가 (리뷰 일관성, 반려율 합리성)
```

**VP 역할 변화:**
- Phase 1: 직접 검수자
- Phase 2+: Quality Lead — 리뷰어 온보딩·품질 관리·최종 이의 중재

### 핵심 전제: 부모의 심리적 의지

> **"자식보다 부모가 100배, 1000배 더 심리적 의지가 강하다."**

Harness 사업 모델의 가장 중요한 전제. 구매 결정과 지속 실행 동력은 모두 **부모**에게 있다.

```
부모 불안 → [공감+증거+해결책] → 부모의 결제 결정 → 부모의 지속 실행 → 아이의 변화
```

### 실행 원칙 7가지

1. **부모가 고객이다** — 모든 마케팅·콘텐츠·가격의 1순위 수신자는 부모
2. **이중 공포를 동시에 인정하라** — "AI를 써도 걱정, 안 써도 걱정" 인지 부조화를 먼저 언어화
3. **손실 언어를 먼저 쓰라** — "얻을 것"보다 "잃을 수 있는 것"을 먼저 (Kahneman 손실 회피 × 부모 보호 본능)
4. **역사를 인용하라** — "이런 전환기는 반복됐다. 승자는 기술을 이해하고 의도적으로 선택했다"
5. **메가스터디가 말 못하는 것을 말하라** — "AI 의존은 조심해야 합니다, 연구가 있습니다"
6. **인구 절벽을 기회로** — "자녀가 하나라면, 더 신중해야 합니다"
7. **도덕적 공황 속의 침착한 가이드** — "우리가 데이터를 보여드리겠습니다"

> **월 100스텝 = 구독자 15명 × 7스텝** or **신규 유저 50명 × 2스텝**
> 박리다매가 작동하면 BEP는 구독자 수가 아니라 "스텝 소비량"으로 측정한다.

### 입소문 설계 원칙

```
입소문 조건:
1. "이게 이렇게 싸?" — 가격 충격 (₩990/스텝)
2. "나한테 딱 맞는 말이었어" — Segment of One 체험 (포춘텔러 효과)
3. "친구한테 보내기 쉬운 링크" — Step 결과 공유 기능

→ 카카오톡으로 Step 결과를 공유하면 친구에게 "Step 1 무료 링크" 제공
→ 바이럴 계수(K-factor) 목표: K ≥ 0.3 (100명 → 130명)
```

**D — 커뮤니티+데이터 누적 루프:**
- 사용자 Step 완료 반응 → LLM 프롬프트 개선 → 다음 사용자 정확도 향상
- 사용자가 많을수록 서비스 품질 향상 (네트워크 효과)

### 3-5. 고마찰 구조 해소 (G-5 해소)

## 5. Codex에게 묻는 구체적 질문

아래 항목들에 대해 **독립적 시각**으로 판단해주세요.

### Q1. 수정된 모델이 Claude/Gemini 이슈를 충분히 해소했는가?

각 BLOCK 항목별로:
- 해소됨 / 부분 해소 / 해소 안됨 중 하나로 평가
- 부분 해소/미해소라면 구체적으로 무엇이 부족한가

### Q2. AI 90% + VP 10% 모델의 VP bottleneck 위험

- 구독자 50명, 100명, 300명 시점에 VP 검수가 실제 병목이 되는가
- 비동기 배치 처리가 가능한가 (Step 완료 → 24시간 내 검수 기준)
- VP가 학부모 언어 전문가로 성장하는 OJT 기간 동안의 품질 공백 관리 방법

### Q3. Step 9~11 (3개월 추적 코칭) LLM 비용 vs 수익 BEP

현재 고정비: LLM API $60~100/월

- Step 9~11 추가 시 LLM 비용 증가분 추정 (각 Step 처리 비용)
- 추적 코칭 Step의 LLM 비용이 수익(₩2,990~₩3,990)을 역전하는 최소 구독자 수
- 3개월 추적이 lock-in으로 작동하기 위한 선행 조건

### Q4. 독점 데이터 확보 경로의 현실성

- A (맘카페 정성 분석): 네이버 ToS §9 자동화 크롤 금지. 수동 수집은 확장성 한계. 실제 독점 데이터로 작동하는가?
- B (학군 AI 수용도 데이터): 이 데이터를 어떻게 수집하는가? 현실적 경로가 있는가?
- D (사용자 반응 데이터): 초기 구독자 0명 시점에는 작동하지 않음. Cold start 문제를 어떻게 해결하는가?

### Q5. 직장인 채널 병행이 초기 집중도를 약화시키는가

- Track A + Track B 동시 운영 시 콘텐츠 생산 부담 (LLM 처리 비용, VP 검수 시간)
- 두 채널을 동시에 광고할 경우 CAC(고객 획득 비용)가 분산되는가
- Gemini의 "타겟 파편화" 지적이 실제로 해소됐는가, 아니면 합리화인가

### Q6. 법적 리스크 (Claude CONCERN-⑥ 미해결)

Claude가 지적했으나 CEO 결정에서 다루지 않은 항목:

- "AI 교육 처방" 서비스의 표시광고법 리스크 (과장 광고)
- "자녀 학습 진단 리포트"의 의료/교육 전문직 유사 행위 여부
- PIPA(개인정보보호법): 음성 온보딩, 자녀 정보 수집 시 동의 설계
- 이 리스크들이 `legal_review_approve` 이전에 사전 검토되었는가
