# EDU 디지털 트윈 시뮬레이션 게이트

> 작성일: 2026-06-13  
> 목적: 실제 대규모 사용자 테스트 전에, 다양한 연령층과 역할의 가상 고객을 LLM 기반 `디지털 트윈`으로 구성해 비가시적 `dark factory` 방식의 full simulation을 반복 실행하고, 점수화된 결과가 기준을 넘을 때만 최종 UX를 구체화·확장한다.

---

## 1. 왜 필요한가

edu 서비스는 단일 사용자용 챗봇이 아니라 다음을 다룬다.

- 부모가 자녀를 리드하는 상황
- 성인 자녀가 노부모를 리드하는 상황
- 직장인이 본인을 리드하는 상황
- 배우자/가족이 함께 의사결정하는 상황
- 모바일 입구 → PC/Mac 심화 전환
- 민감한 상담 기록과 매직 링크 복구

이런 구조는 소수 수동 테스트만으로는 약점을 충분히 찾기 어렵다.

따라서 실제 대규모 공개 전에는:

1. 다양한 가상 고객군을 만든다.
2. 서로 문답을 주고받는 다중 에이전트 시뮬레이션을 돌린다.
3. 결과만 수집해 점수화한다.
4. RED TEAM 기준점 이상을 넘지 못하면 UX를 확정하지 않는다.

## 2. 기본 개념

### 2.1 디지털 트윈

실제 고객군을 대표하는 가상 인물 세트.

예:

- 중2 자녀를 둔 40대 학부모
- 30대 직장인 자녀, 70대 부모 돌봄 책임자
- 50대 비개발 직장인, 회사 AI KPI 압박 대상
- 스마트폰만 쓰는 70대 노인
- 노트북 위주로 과제를 하는 대학생

각 트윈은 최소 아래 속성을 가진다.

- 연령대
- 성별
- seeker 역할
- target 역할
- 디지털 숙련도
- 기기 선호도
- AI 불안 유형
- 결제 저항
- 개인정보 민감도
- 이탈 패턴
- 선택 LLM 성향
- 최신 트렌드 반응성

중요:

- 트윈은 임의로 지은 캐릭터가 아니라, 기존에 수집한 맘카페, 블로그, 유튜브, RSS, 커뮤니티 발화 패턴을 반영한 `evidence-grounded twin`이어야 한다.
- 말투, 질문 방식, 불안 표현, 이탈 표현, 거부감 표현도 source corpus 기반으로 만들어야 한다.

### 2.2 Dark Factory simulation

실제 UI를 사람이 보는 방식이 아니라, 백엔드/정책/문구/상태전이를 LLM 에이전트들이 반복 시뮬레이션하는 비가시적 테스트 방식.

핵심:

- 수백~수천 회 반복 가능
- 동일 시나리오 재현 가능
- 실패 패턴 자동 축적 가능
- 실제 공개 전 취약점 선제 발견 가능

## 3. 언제 실행하는가

아래 변경은 simulation gate를 반드시 통과해야 한다.

1. 공개 UX 구조 변경
2. intake 필드 변경
3. 매직 링크/재방문/복구 로직 변경
4. 모바일→데스크톱 handoff 로직 변경
5. 연령대/성별/가족관계/target 모델 변경
6. offer timing / 유료 전환 구조 변경
7. 민감정보/권한/소유자 확인 정책 변경

즉, `고객 경험`, `전환`, `보안`, `가족 리드 구조`에 영향을 주는 변경은 simulation gate 없이 확정하지 않는다.

## 4. 시뮬레이션 구성

### 4.1 에이전트 역할

최소 아래 역할을 둔다.

- `customer twin agent`
  - 특정 연령/역할/디지털 숙련도를 가진 가상 고객
- `service agent`
  - 실제 제품 로직 또는 그것을 모사한 응답기
- `observer agent`
  - 상태전이, 이탈, 혼란, 보안 위험 관찰
- `red team scorer`
  - fail/pass 판정과 점수화

### 4.2 필수 시나리오 축

각 시나리오는 아래 축을 조합해 만든다.

- seeker 연령대
- seeker 성별
- target 종류
- target 연령대
- 스마트폰 only / PC available / cross-device
- 카카오 인앱 브라우저 / 일반 모바일 브라우저 / 데스크톱 브라우저
- 새로 시작 / 이어보기 / 만료 링크 / 잘못된 링크
- 연락처 입력 전 이탈 / 입력 후 이탈 / 데스크톱 전환 직전 이탈
- 결제자와 실제 사용자가 동일 / 상이
- 선택한 LLM 종류
- 최신 트렌드/버전 차이에 민감한 사용자 vs 둔감한 사용자
- `교과서형 과제에 거부감을 보이는 사용자` vs `실제 결과물 실습에 반응하는 사용자`

### 4.3 최소 시나리오 세트

최소 아래는 항상 포함한다.

1. 40대 학부모, 카카오톡 링크 진입, 자녀 문제 상담
2. 30대 성인 자녀, 노부모 대신 진입, 대리 리드 구조
3. 50대 직장인, 모바일에서 시작 후 PC 실습 전환
4. 70대 노인, 모바일 only, 긴 입력에 취약
5. 같은 케이스를 모바일과 PC에서 동시에 여는 충돌 시나리오
6. 잘못 공유된 복구 링크를 타인이 열려는 보안 시나리오

## 5. 점수화 체계

각 시뮬레이션은 아래 항목을 최소 점수화한다.

### 5.1 UX 점수

- 시작 성공률
- 1분 내 핵심 정보 입력률
- 혼란 신호 빈도
- 모바일 가독성/행동 가능성
- 데스크톱 전환 성공률

### 5.2 안전/보안 점수

- 잘못된 링크 접근 차단률
- 타인 케이스 노출 위험
- 다중 기기 충돌 발생률
- 민감정보 과다 노출 여부

### 5.3 상담 품질 점수

- seeker/target 구분 유지
- 연령/성별/숙련도 맥락 반영
- 과도한 추정 여부
- 실행 과제의 현실성
- 리드 가능한 방향 제시 여부
- 교과서형/형식적 처방 비율
- 실제 결과물 생성 유도력
- 고객이 선택한 LLM과 가이드의 일치율

### 5.3A 트렌드/툴 적합성 점수

- 최신 LLM/버전 흐름 반영 여부
- 선택한 툴과 무관한 설명이 끼어드는 비율
- 설치/실행 가이드의 현재성
- 최신 인터넷/유튜브 기반 사용법 변화 반영 여부

### 5.4 전환/운영 점수

- 이어보기 복구 성공률
- 이탈 후 재진입 복구 성공률
- operator가 원인 파악 가능한 수준의 로그 확보 여부
- offer timing 적절성

## 6. RED TEAM 게이트

최종 UX는 `simulation 결과 + RED TEAM 점수`가 기준을 넘기 전에는 구체화/확장하지 않는다.

### 6.1 판정 원칙

- `block`
  - 개인정보 노출, 잘못된 케이스 접근, 동시성 충돌, seeker/target 혼동이 치명적일 때
- `needs_work`
  - UX 혼란, 낮은 handoff 성공률, 낮은 복구 성공률, 낮은 operator 관측성이 있을 때
- `clear`
  - 핵심 보안/복구/전환/상담 품질 기준을 모두 넘을 때

### 6.2 운영 규칙

1. RED TEAM 기준점 미달이면 실제 대규모 공개/확장 금지
2. 점수 미달 항목은 backlog가 아니라 release blocker로 기록
3. 최종 UX 세부화는 `simulation gate pass` 이후에만 진행

## 7. 구현 전제

이 simulation은 처음부터 완전한 화면 렌더링 기반일 필요는 없다.

초기에는 다음 순서로 간다.

1. 텍스트 상태전이 기반 simulation
2. API contract 기반 simulation
3. 매직 링크/복구/handoff 로직 simulation
4. 필요 시 브라우저 자동화 기반 UI simulation 추가

또한 simulation에 들어가는 지식은 정적 문서로 고정하면 안 된다.

- 최신 LLM 기능/버전 차이
- 최근 유튜브 클립/튜토리얼
- 커뮤니티에서 실제로 쓰는 표현
- 최신 사용성 변화

를 주기적으로 수집·정제해 twin과 scorer가 참조할 수 있어야 한다.

즉, 먼저 `보이지 않는 dark factory`를 만들고, 그 결과를 가지고 실제 UX를 다듬는다.

## 9. 현재 결정

현재 edu 제품군에 대해 아래를 기본 규칙으로 확정한다.

1. PoC는 몇 차례 직접 수동 테스트를 수행한다.
2. 대규모 시험 전에는 다양한 연령층 기반 `디지털 트윈` full simulation을 수행한다.
3. simulation은 사람이 화면을 보며 누르는 테스트보다 먼저 도는 `dark factory` 방식으로 구축한다.
4. 최종 UX는 simulation 기반 RED TEAM 합격 점수 이상을 획득한 뒤에만 구체화한다.

이 문서는 향후 edu UX, case orchestration, red-team harness, operator scoring pipeline의 상위 게이트 규칙으로 사용한다.
