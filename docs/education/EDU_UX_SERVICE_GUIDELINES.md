# EDU 상담 서비스 UX 설계 지침

> 작성일: 2026-06-13  
> 목적: Harness edu 사업을 `부모 1호 파일럿`에서 멈추지 않고, 장기적으로 `전 연령 AI 불안/학습/리드 상담 서비스`로 확장할 때 흔들리지 않도록 고객 UX 원칙을 고정한다.

---

## 1. 제품의 실제 진입점

고객의 첫 진입은 대부분 검색창이 아니라 아래 중 하나다.

1. 카카오톡 메시지 안의 매직 링크
2. 지인이 전달한 카카오톡 공유 링크
3. 문자/이메일로 받은 이어하기 링크
4. 모바일 브라우저에서 열린 광고/랜딩의 CTA

즉, 기본 가정은 `데스크톱 웹 첫 방문`이 아니라 `스마트폰 메신저 안에서 링크를 누르는 첫 방문`이다.

따라서 edu UX는 아래를 기본값으로 가져야 한다.

- 첫 진입은 모바일 세로 화면 기준으로 설계한다.
- 첫 행동은 긴 입력이 아니라 `짧은 선택 + 한 줄 답변` 중심이어야 한다.
- 첫 진입에서 seeker의 기본 demographic을 명시적으로 받는다.
  - 연령대
  - 성별
- 계정 생성보다 `바로 시작`이 먼저여야 한다.
- 재방문 복구는 로그인보다 `매직 링크 + 케이스 이어보기`가 먼저여야 한다.

여기서 중요한 원칙은 다음과 같다.

- 연령/성별은 `추정`하지 않는다.
- 사용자가 직접 선택한 값만 사용한다.
- 응답 거부/비공개 선택지를 항상 둔다.
- 기기/환경도 `추정`하지 않는다.
- iPhone/Android/Windows/Mac 여부는 사용자가 직접 고르게 한다.

## 2. 고객 변화 모델

Harness가 만드는 변화는 단순 정보 제공이 아니다.

1. 고객이 본인의 AI 불안이나 무력감을 언어로 설명할 수 있게 한다.
2. 고객이 먼저 AI를 배워서 `상대적 전문가`가 되게 한다.
3. 그 고객이 자녀, 부모, 배우자, 팀원처럼 본인이 걱정하는 상대를 리드할 수 있게 한다.

따라서 상담 UX의 핵심 객체는 단순 `segment` 하나가 아니라 아래 셋이다.

- `seeker`: 지금 상담을 받는 사람
- `target_person`: seeker가 걱정하거나 리드하려는 상대
- `goal`: seeker가 얻고 싶은 변화

예시:

- seeker=학부모 / target_person=중2 자녀 / goal=과제용 AI 의존 줄이기
- seeker=직장인 자녀 / target_person=노부모 / goal=AI 사기/디지털 소외 예방
- seeker=직장인 / target_person=본인 / goal=업무 AI 숙련

이 구조가 없으면 장기적으로 전 연령 상담 서비스로 확장할 수 없다.

### 2.1 교과서형 상담을 금지한다

고객은 `모범 답안 같은 부모 역할극`을 원하지 않는다.

따라서 아래 같은 과제는 기본값으로 제시하면 안 된다.

- `자녀에게 AI를 학습에 어떻게 활용하고 있니? 라고 물어보세요`
- `자녀와 함께 AI로 문제를 풀고 토론해보세요`

이런 문구는 맞는 말처럼 보여도 실제 부모의 현재 숙련도와 감정 상태를 무시한 `교과서형 처방`으로 받아들여지기 쉽다.

우리가 줘야 하는 것은:

1. 지금 왜 이 문제가 뜨거운지에 대한 현실적 맥락
2. 최근 트렌드를 공격적으로 짚는 짧고 선명한 설명
3. `내가 지금 이 수준이어도 시작할 수 있다`는 감각
4. 몇 주 안에 `준전문가`로 올라갈 수 있다는 실제 경로

즉, 상담은 훈계나 이상론이 아니라 `흥미를 유발하는 현실형 가이드`여야 한다.

## 3. 디바이스 전략 원칙

기기 전략은 `모바일` / `데스크톱` 2분법으로 끝나면 안 된다.

실제 안내는 아래 환경 차이를 반영해야 한다.

- iPhone
- Android
- Windows PC
- Mac
- 인앱 브라우저 / 일반 브라우저
- 앱 설치 가능 / 설치 제한
- 혼자 진행 / 옆에서 도와줄 사람 있음

### 3.1 모바일은 입구다

모바일에서 가장 잘해야 하는 일:

- 바로 시작
- 짧은 자가점검
- 케이스 생성
- 핵심 불안/상황 파악
- 1~3개의 즉시 실행 과제 제시
- 매직 링크 재방문 복구
- 가족/지인에게 결과 공유

즉시 실행 과제도 아래 기준을 지켜야 한다.

- 고객의 현재 숙련도보다 한 단계만 앞서야 한다.
- 부모가 아직 AI를 잘 모르면 `자녀와 대화하라`보다 `부모 본인이 먼저 눈이 뜨이는 경험`을 준다.
- 과제는 설명이 아니라 실제 산출물을 만들게 해야 한다.
- `5분 안에 해본 결과가 눈에 보이는 경험`이 먼저 와야 한다.

모바일에서도 반드시 구분해야 한다.

- `iPhone인지 Android인지`
- `카카오 인앱인지 사파리/크롬인지`
- `앱 설치에 익숙한지`

### 3.2 PC/Mac는 심화 구간이다

난이도가 올라갈수록 고객은 큰 화면과 키보드가 필요하다.

PC/Mac에서 가장 잘해야 하는 일:

- 긴 문서 읽기
- 여러 탭을 띄우고 비교하기
- 프롬프트 복사/편집
- 파일 업로드/다운로드
- 실제 AI 도구를 켜고 따라하기
- 장문의 reflection 작성
- 가정/직장용 가이드 문서 정리

여기서 중요한 점은, 많은 고객이 `PC/Mac 심화 구간`에 들어오더라도 실제로는 초보자라는 것이다.

- 어떤 앱을 설치해야 하는지 모른다.
- 설치는 했지만 어디서 실행하는지 모른다.
- 실행은 했지만 무엇을 눌러야 하는지 모른다.
- Claude Desktop, ChatGPT 앱, 브라우저 탭, 파일 탐색기 같은 기본 조작도 막힐 수 있다.

따라서 `도구를 설치하고 켜는 과정` 자체를 상담 제품의 일부로 취급해야 한다.

그리고 여기서도 `Windows인지 Mac인지`에 따라 안내가 갈라져야 한다.

- 앱 설치 위치가 다르다
- 다운로드 후 실행 방식이 다르다
- 보안 경고/권한 승인 흐름이 다르다
- 파일 찾는 방식이 다르다

추가로, 심화 실습은 `고객이 선택한 LLM`을 기준으로 갈라져야 한다.

- Claude를 고른 고객에게 Gemini 교육을 하면 안 된다.
- Gemini를 고른 고객에게 Claude Desktop 기준 설명을 하면 안 된다.
- Codex, Claude Code, Gemini, ChatGPT 등은 설치/로그인/첫 실행 흐름이 모두 다르다.

우리는 추천은 할 수 있지만, 실제 실습 가이드는 반드시 `고객이 선택한 툴` 기준이어야 한다.

### 3.3 제품은 디바이스 전환을 설계해야 한다

좋은 UX는 고객이 알아서 PC로 옮겨가길 기대하지 않는다.

반드시 제공해야 하는 것:

1. `이 단계는 휴대폰에서 3분이면 됩니다`
2. `이 다음 단계는 PC/Mac에서 하시면 더 쉽습니다`
3. `내게 링크 보내기`
4. `이어서 하기`
5. `지금까지 한 내용 요약`
6. `설치가 필요한 경우 어디서 막히는지 한 단계씩 안내`
7. `실행까지 완료됐는지 확인하는 체크포인트`

## 4. 매직 링크 설계 원칙

매직 링크는 편의 기능이 아니라 `주 인증/복구 수단`에 가깝게 취급해야 한다.

### 4.0 링크 종류를 분리한다

링크는 최소 두 종류여야 한다.

1. `공유/홍보 링크`
   - 랜딩 또는 공개 시작 화면으로만 이동
   - 개인 상담 기록을 절대 열지 않음
2. `개인 복구 링크`
   - 특정 case를 여는 비공개 링크
   - 만료 시간, 재발급, 소유자 확인 정책을 가짐

이 둘을 섞으면 `공유는 쉬운데 개인정보는 새나가는` 최악의 구조가 된다.

### 4.1 반드시 보장해야 하는 것

1. 링크는 `특정 customer`가 아니라 `특정 case`를 열어야 한다.
2. 같은 사람이 여러 케이스를 가질 수 있어야 한다.
3. 새로 시작과 이어보기가 섞이면 안 된다.
4. 링크를 눌렀을 때 `왜 이 화면이 열렸는지`가 설명돼야 한다.
5. 링크 만료/사용 완료 시 복구 경로가 있어야 한다.
6. 개인 복구 링크는 `공유용 CTA`와 같은 위치/카피로 노출하지 않는다.
7. 민감 상담 내용을 여는 링크는 최소한의 소유자 재확인 절차를 가진다.

### 4.1A 소유자 재확인 원칙

민감한 케이스를 여는 복구 링크는 아래 중 최소 하나를 추가 검토해야 한다.

- 이메일 일회용 코드
- 전화번호 뒷자리 또는 생년월일 일부 확인
- 같은 기기/브라우저에서만 즉시 열고, 새 기기에서는 2차 확인

초기 Pretotyping에서는 마찰이 늘어날 수 있지만, 가족/자녀/노인 관련 민감 대화는 `링크만 알면 열리는 구조`로 오래 운영하면 안 된다.

### 4.2 반드시 피해야 하는 것

- 같은 이메일이라는 이유로 무조건 최신 케이스 열기
- 예전 상담을 조용히 이어붙이기
- 사용자가 새 상담이라고 생각하는데 시스템은 이어보기로 동작하기
- 만료된 링크에서 막다른 에러만 보여주기

### 4.3 반드시 추가해야 하는 것

- 최근 케이스 목록 1~3개
- 마지막 상담 시각
- 한 줄 요약
- `이 케이스 이어보기` / `새 케이스 시작`
- `이 링크를 다른 기기로 보내기`
- `링크가 열지 않는 경우 다시 받기`
- `이 링크는 개인 기록을 여는 링크`라는 경고

## 5. 카카오톡/모바일 브라우저 특화 리스크

카카오 인앱 브라우저나 모바일 브라우저는 데스크톱과 다르게 자주 깨진다.

### 5.1 실제로 자주 생길 문제

- 클립보드 복사 권한 제한
- 새 창 열기 제한
- 세션/스토리지 보존 불안정
- 긴 텍스트 입력 피로
- 파일 다운로드 위치를 사용자가 모름
- 브라우저를 닫으면 맥락 상실
- 키보드가 화면을 가려 입력창/버튼이 안 보임
- 스크롤 하단 CTA를 못 보고 이탈

### 5.2 설계 원칙

- 중요한 상태는 localStorage만 믿지 말고 서버 케이스에도 남긴다.
- 다운로드보다 `링크 보내기`, `카톡으로 이어받기`, `나중에 PC에서 열기`를 우선한다.
- 모바일에서는 긴 PDF보다 짧은 실행 카드가 먼저다.
- 모든 핵심 액션은 화면 하단 고정 또는 마지막 메시지 근처에 둔다.
- 클립보드/새창이 실패해도 대체 경로를 둔다.
- 인앱 브라우저에서 막히면 `사파리/크롬으로 열기` fallback을 제공한다.
- 링크 재발급 실패, 새창 차단, 다운로드 실패를 operator가 추적할 수 있어야 한다.

## 6. 언제 PC/Mac 전환을 권해야 하는가

아래 상황에서는 시스템이 명시적으로 큰 화면 전환을 권해야 한다.

1. 프롬프트 복사 후 실제 툴에서 실행해야 할 때
2. 여러 단계의 비교/검토가 필요할 때
3. 파일 업로드가 필요한 과제일 때
4. 가정 규칙/업무 규칙 문서를 작성해야 할 때
5. 부모가 자녀/부모와 함께 화면을 보며 대화해야 할 때
6. 노인 대상일 때, 자녀가 대신 PC에서 세팅해 줄 필요가 있을 때

권장 문구 예:

- `여기부터는 휴대폰보다 PC/Mac에서 하시면 훨씬 편합니다. 지금 링크를 보내드릴게요.`
- `오늘은 모바일에서 점검만 마치고, 심화 실습은 큰 화면에서 이어가세요.`

금지해야 할 안내 방식:

- `플레이스토어에서 앱 검색 후 설치하세요`
- `Claude Desktop 설치 후 실행하세요`
- `브라우저를 열고 로그인하세요`

이런 dry 문구는 초보자에게는 행동 지시가 아니라 이탈 유도문에 가깝다.

반드시 아래처럼 바꿔야 한다.

1. 지금 필요한 이유를 먼저 말한다.
2. 지금 어디까지 왔는지 보여준다.
3. 다음 행동을 1개만 제시한다.
4. 막히는 대표 지점을 미리 알려준다.
5. 성공했는지 바로 확인하게 한다.

예:

- `이제 큰 화면에서 같이 해보면 훨씬 쉽습니다. 먼저 PC나 Mac에서 이메일만 열어주세요. 제가 이어서 할 링크를 보내드릴게요.`
- `링크를 여셨다면 화면 아래 파란 버튼 하나만 눌러보세요. 설치 화면이 안 보이면 제가 다른 방법으로 안내할게요.`
- `앱이 열리면 성공입니다. 아직 질문은 하지 말고, 열린 화면만 확인해 주세요.`

또한 과제 제시는 아래처럼 바뀌어야 한다.

- `자녀에게 물어보세요`보다 `부모가 먼저 써보고 바로 결과를 보게 하는 것`
- `토론해보세요`보다 `지금 원하는 방향의 결과물을 LLM으로 직접 만들어보는 것`

예:

- `지금 Claude에서 "중2 아이가 AI로 숙제를 베껴 쓰는지 알아차리는 부모 체크포인트 5개"를 만들어보게 해드릴게요. 결과를 보면 바로 감이 옵니다.`

## 7. 고객 유형별 디바이스 차이

고객 유형뿐 아니라 `기기 환경 프로파일`도 함께 받아야 한다.

필수 수집 후보:

- 현재 기기 종류
- 현재 OS
- 나중에 사용할 큰 화면 기기 종류
- 주 브라우저
- 앱 설치 가능 여부
- 회사/학교 보안 제한 여부
- 옆에서 도와줄 사람 유무

### 7.1 학부모

- 최초 진입: 카카오톡 링크
- 초기 행동: 짧은 답변, 자녀 상황 설명
- 심화 전환: 자녀와 함께 PC/Mac에서 실습, 가정 규칙 문서화

### 7.2 직장인

- 최초 진입: 모바일 가능
- 진짜 가치 구간: PC/Mac에서 업무 도구와 같이 실습
- 모바일만으로 끝나는 상품이 되면 안 된다
- 설치/실행/로그인/파일 접근에서 가장 많이 막힐 수 있으므로, `업무 실습`보다 먼저 `도구 준비 완료` 경험을 만들어야 한다

### 7.3 노인

- 본인 직접 진입보다 `자녀 대리 진입` 비율이 높을 수 있다
- 스마트폰에서 너무 복잡하면 바로 이탈한다
- 자녀가 먼저 받고, 이후 노인에게 보여주거나 같이 실행하는 2인 UX가 필요하다

### 7.4 대학생/청소년

- 모바일 진입이 자연스럽지만, 실제 과제/학습 환경은 노트북 비중이 높다
- 상담 결과를 `오늘 바로 해볼 것`과 `노트북에서 실습할 것`으로 분리해야 한다

## 8. 반드시 구현해야 할 UX 기능

### 8.1 모바일 first 필수

- 1탭 진입
- 1분 안에 시작 가능
- 1문장씩 답해도 진행 가능
- seeker의 연령대/성별을 초반 1~2탭 안에 명시 입력
- seeker의 현재 기기 환경을 초반 intake에서 짧게 입력
  - iPhone / Android
  - 인앱 브라우저 / 일반 브라우저
- 매직 링크 복구
- 마지막 상태 서버 저장
- `PC/Mac에서 이어서 하기`를 눌렀을 때 설치/실행 onboarding으로 자연스럽게 이어짐

### 8.2 cross-device 필수

- `내게 이어하기 링크 보내기`
- `PC에서 이어하기`
- `지금까지 요약 보기`
- `다음 단계는 큰 화면 권장` 안내
- `모바일 세션 잠금/해제` 또는 `다른 기기에서 열려 있음` 표시
- `같은 케이스를 두 기기에서 동시에 수정하지 않도록` 동시성 제어
- 설치 필요 시 `설치 전`, `설치 중`, `실행 완료` 상태를 단계별로 추적
- `어디서 막혔는지`를 사용자가 한 번 탭해서 표시할 수 있어야 함
- 큰 화면 기기 종류를 묻고 Windows/Mac 분기 가이드를 제공
- 회사/학교 보안 제한이 있으면 브라우저 버전 대체 경로 제공
- 고객이 선택한 LLM별 설치/실행/첫 과업 흐름 분기
- `추천 LLM`과 `선택한 LLM`을 구분 저장

### 8.3 operator 필수

- 고객이 어느 기기에서 들어왔는지 추적
- 모바일에서 멈췄는지, 데스크톱으로 넘어갔는지 추적
- 어느 단계에서 이탈하는지 퍼널 기록
- 매직 링크 실패율 기록
- 인앱 브라우저/일반 브라우저 구분 기록
- 링크 재발급 횟수, 만료 횟수, 소유자 확인 실패 횟수 기록
- 결제자와 실제 상담 참여자가 다른 경우를 식별
- 설치 시작률 / 설치 완료율 / 첫 실행 완료율 / 첫 과업 시작률을 별도 추적
- 고객이 `설치`, `로그인`, `앱 실행`, `파일 선택`, `복사/붙여넣기` 중 어디서 막히는지 기록
- iPhone/Android/Windows/Mac별 성공률과 이탈률을 분리 추적
- 인앱 브라우저/일반 브라우저별 실패 패턴을 분리 추적

## 9. 현재 설계의 부족한 점 체크리스트

현재 Harness edu UX가 특히 조심해야 할 부족점:

1. 부모/직장인 2세그먼트에 과도하게 고정됨
2. seeker와 target_person 분리 모델 부재
3. 모바일 입구는 있으나 `PC/Mac handoff UX`가 없음
4. 매직 링크는 열리지만 최근 케이스 선택 UX가 얕음
5. 결과물 다운로드는 있으나 `다른 기기로 보내기`가 없음
6. 모바일에서 끝내도 되는 단계와 큰 화면이 필요한 단계를 구분하지 않음
7. 카카오 인앱 브라우저 실패 패턴을 전제로 한 fallback 설계가 약함
8. 노인/대리결제/가족 동행 사용 시나리오가 제품 객체로 정의되지 않음
9. 상담 결과가 `실행 카드`보다 `채팅 기록`에 더 묶여 있음
10. 고객이 “내가 먼저 배우고 상대를 리드한다”는 구조가 명시적으로 드러나지 않음

## 10. 앞으로 문서와 코드가 따라야 할 규칙

향후 edu 관련 신규 문서/코드/UX는 아래를 기본값으로 삼는다.

1. 최초 진입은 모바일 매직 링크를 기준으로 설계한다.
2. 심화 학습은 PC/Mac 전환을 전제로 설계한다.
3. `segment`만으로 고객을 설명하지 말고 `seeker / target_person / goal`을 분리한다.
4. 매직 링크는 항상 특정 케이스를 정확히 연다.
5. 새로 시작과 이어보기는 절대 혼동되지 않게 한다.
6. 모바일에서는 짧은 실행 카드, 데스크톱에서는 심화 실습/문서화를 우선한다.
7. 카카오/모바일 브라우저 제약을 실패 시나리오가 아니라 기본 환경으로 가정한다.
8. 고객이 다른 가족 구성원을 리드하게 만드는 구조를 상품 중심에 둔다.
9. 공유 링크와 민감 기록 복구 링크를 절대 같은 성격으로 다루지 않는다.
10. 동일 케이스의 다중 기기 동시 수정 충돌을 제품 레벨에서 제어한다.
11. 대규모 공개 전에는 수동 테스트만이 아니라 `디지털 트윈 / dark factory simulation gate`를 통과해야 한다.
12. 설치/실행/로그인/첫 실습은 부가 설명이 아니라 핵심 UX로 설계한다.
13. 기기와 주변 환경은 추정하지 말고 직접 물어본 뒤 그 값에 맞는 분기 가이드를 제공한다.
14. 고객 과제는 교과서형 질문이 아니라 `흥미를 유발하고 눈에 보이는 산출물을 만드는 실습`이어야 한다.
15. 실습 가이드는 반드시 고객이 선택한 LLM 기준으로 맞춤 제공한다.

이 문서는 향후 edu 공개 UX, PWA, 네이티브 앱, 카카오 채널 자동화, operator dashboard 설계의 공통 상위 지침으로 사용한다.

simulation gate의 상세 규칙은 [EDU_SIMULATION_GATING.md](EDU_SIMULATION_GATING.md)를 따른다.
