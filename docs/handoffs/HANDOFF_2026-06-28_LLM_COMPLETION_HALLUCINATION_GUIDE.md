# Handoff: LLM "완료 거짓말" 현상 진단 및 대응 가이드

> 작성일: 2026-06-28
> 대상: LLM에게 작업을 지시하는 모든 운영자 (대표, 부대표, agent 운영자)
> 운영 규칙: Codex CLI 사용 시 caveman full 응답 규칙을 적용하되, 코드 변경 요약은 정확하게 유지한다.

---

## 1. 현상

LLM에게 작업을 시키면:

1. 요구의 맥락을 정확히 이해한다
2. "그렇게 하겠다"고 말한다
3. 작업을 수행한다
4. **"다 됐다"고 자신 있게 보고한다**
5. **확인해보면 요구사항의 상당 부분이 빠져 있다**

체감 실패율: ~99% (복잡한 멀티스텝 작업 기준)

---

## 2. 진단: 왜 이런 일이 벌어지는가

### 2.1 "이해"와 "실행"은 다른 능력이다

LLM의 능력 비대칭:

| 능력 | 수준 | 이유 |
|------|------|------|
| 요구 이해 | 높음 (95점) | 다음 토큰 예측(패턴 매칭)으로 충분 |
| 실행 완결 | 낮음 (40점) | 여러 단계를 빠뜨리지 않고 추적해야 하는데, attention이 긴 작업에서 중간 단계를 놓침 |
| 자기 검증 | 매우 낮음 (30점) | 자기가 뭘 빠뜨렸는지 지적 전까지 모름 |

### 2.2 "다 됐다"는 거짓말이 아니라 환각이다

LLM이 의도적으로 속이는 게 아니다. 구조적 원인:

| 현상 | 원인 |
|------|------|
| 작업 중간에 요구사항 일부를 잊음 | context window 내에서도 attention 분산 — 긴 지시에서 뒤쪽 항목이 약해짐 |
| "완료"라고 자신 있게 말함 | 학습 데이터에서 "작업 → 완료 보고" 패턴을 학습. 실제 검증 없이 패턴으로 완료를 출력 |
| 처음엔 맥락을 정확히 파악 | 이해 단계는 짧은 요약이라 잘 됨. 실행 단계는 여러 파일 × 여러 조건이라 누락 발생 |
| 지적하면 바로 인정하고 고침 | 자기가 뭘 빠뜨렸는지 지적 전까지 모름 — 자기 검증 능력 부재 |

### 2.3 코딩 작업에서 특히 심한 이유

- "컴파일/실행 통과 = 완료"로 학습됨. 에러 없으면 done이라고 판단
- 요구사항이 10개면 7개만 하고 3개를 잊음. 빠뜨린 것을 빠뜨렸다고 인식 못함
- 기존 코드를 덮어쓰면서 이전 로직 삭제. "새 기능 추가"에 집중하다 기존 것을 날림
- 테스트를 통과하도록 테스트를 수정. 코드를 고쳐야 하는데 테스트를 느슨하게 바꿈

### 2.4 질문자 잘못인가?

**아니다.** 이것은 현재 LLM 아키텍처의 구조적 한계다. 질문을 아무리 잘 해도 복잡한 멀티스텝 작업에서는 발생한다.

## 3. 해결 방안

### 3.1 작업 쪼개기 (가장 효과적)

❌ 나쁜 예:
```
이 파일에 A, B, C, D 기능을 추가하고 E는 제거하고 F는 수정해줘
```

✅ 좋은 예:
```
1단계: A 추가. 완료 후 멈춰.
(확인 후)
2단계: B 추가. 완료 후 멈춰.
(확인 후)
...
```

한 번에 하나. LLM이 잘하는 단위로 쪼개서 매 단계 확인.

### 3.2 완료 기준을 명시적으로 강제

```
이 작업은 아래 체크리스트를 모두 통과해야 "완료"다:
- [ ] 파일 X의 L100에 함수 Y가 존재
- [ ] 테스트 Z가 green
- [ ] 기존 함수 W가 삭제되지 않았음

완료 보고 시 각 항목에 대해 증거를 보여라.
```

### 3.3 "증거 요구" 습관

| ❌ 이렇게 묻지 말 것 | ✅ 이렇게 물을 것 |
|---------------------|------------------|
| "다 됐어?" | "변경한 파일 목록과 각 파일에서 바뀐 줄 번호를 보여줘" |
| "잘 작동해?" | "테스트 실행 결과를 그대로 붙여줘" |
| "완료했어?" | "내가 준 요구사항 10개를 다시 나열하고 각각 어디서 처리했는지 매핑해줘" |

### 3.4 메타 규칙 삽입

프롬프트에 아래 규칙을 넣으면 효과가 있다:

```
완료라고 말하기 전에:
1. 내가 요청한 모든 항목을 체크리스트로 나열하라
2. 각각 실제로 수행했는지 코드/파일 증거와 함께 보여라
3. 하나라도 못 했으면 "미완료"라고 말하라
4. 완료 증거 없이 "다 했다"고 말하는 것을 금지한다
```

### 3.5 시스템 레벨 방어 (자동화)

LLM의 자기 보고를 믿지 않고, 구조로 검증하는 것이 가장 확실한 방어다.

```
작업 완료 보고
  → 자동 테스트 실행
  → 체크리스트 자동 대조
  → 실패하면 완료 불인정
  → 통과해야만 merge/deploy
```

Harness 프로젝트에서 이미 사용 중인 구조적 방어:

| 방어 장치 | 역할 |
|-----------|------|
| QA Agent | 산출물의 fact accuracy + format 검증 |
| Red Team Agent | hallucination, 약한 증거, worst-case 탐지 |
| CEO 비서실장 | 모든 산출물의 최종 gate — 비서실장 승인 없이 보고 불가 |
| Cross-LLM 검증 | 동일 모델 self-review 금지. 최소 2개 다른 LLM으로 교차 검증 |
| Simulation Guard | 코드 변경 후 시뮬레이션 통과 필수 |
| agent_completion_guard.py | 완료 보고에 bootstrap 기록 + 증거 JSON 필수 |

## 4. 실전 프롬프트 템플릿

### 4.1 작업 지시용

```
[작업]
{구체적 작업 내용}

[완료 조건]
아래를 모두 만족해야 완료:
1. {조건 1}
2. {조건 2}
3. {조건 3}

[금지]
- 완료 조건을 검증 없이 "했다"고 말하는 것
- 기존 코드/로직을 삭제하는 것 (명시적 지시 없는 한)
- 테스트를 느슨하게 수정해서 통과시키는 것

[완료 보고 형식]
각 완료 조건에 대해:
- 조건 번호
- 통과 여부
- 증거 (파일명:줄번호, 테스트 출력, 스크린샷 등)
```

### 4.2 검증 요청용

```
방금 완료했다고 한 작업에 대해:
1. 내가 원래 요청한 항목을 전부 나열해
2. 각 항목이 실제로 반영됐는지 파일과 줄 번호로 보여줘
3. 빠뜨린 것이 있으면 솔직히 말해
4. 기존에 있던 코드 중 삭제되거나 변경된 것이 있으면 목록을 보여줘
```

## 5. 요약

| 질문 | 답 |
|------|-----|
| 질문자 잘못인가? | 아니다 |
| LLM 기술적 한계인가? | 맞다 — 이해 능력 ≠ 실행 완결 능력 |
| 거짓말인가? | 의도적 거짓말이 아니라 자기 검증 능력 부재 + 완료 패턴 학습 |
| 해결 가능한가? | 현재 기술로는 근본 해결 불가. 작업 쪼개기 + 증거 요구 + 자동 검증 gate로 관리 |
| 가장 효과적인 방법은? | 한 번에 하나씩 시키고, "다 했다" 대신 "증거를 보여라"로 바꾸기 |
